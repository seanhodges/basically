# ZX Spectrum 128K / +2 / +3 dialect — implementation plan

> Generated by the `adding-a-target-system` skill on 2026-06-22. Each stage is a
> medium, single-session task for the coding agent. Run them in order; the
> scaffolding under `src/dialects/zxspectrum128/` is throwing stubs (with the
> truly-identical pieces re-exported from `../zxspectrum`) until filled in. The
> dialect is **not** registered until Stage 3.

## Target summary

- **id / name:** `zxspectrum128` / `Spectrum 128K`
- **CPU / bus pattern:** in-tree **Z80 bus** over the vendored `src/emulator/z80/`
  core — an extension of the 48K Spectrum machine in
  `src/dialects/zxspectrum/emulator/`, adding paged memory, a dual ROM and the
  AY-3-8912 register file. This dialect reuses the 48K language and tape layers
  the way `bbcmaster` reuses `bbcmicro`; only the genuinely 128-specific hardware
  and the two extra BASIC tokens are new.
- **One dialect for the family.** 128K, +2 and +2A/+3 all run the same "128
  BASIC". The **128K and +2 are functionally identical** (the +2 is a 128K with a
  built-in tape deck and a patched but token-compatible ROM), so one
  `zxspectrum128` dialect on the 128K ROM covers both. The **+2A/+3** add a
  second paging port (`0x1FFD`), a 4×16K ROM set and the +3DOS disk system —
  out of scope for this 32K-ROM build; noted as a Stage 5 stretch.
- **Display size:** 256×192 — the app default, so `displaySize` is **omitted**.
  The ULA display is byte-for-byte the 48K's; the only addition is the **shadow
  screen** in RAM bank 7, selected by bit 3 of port `0x7FFD` (Stage 2 points the
  renderer at whichever bank is currently displayed).
- **Image / tape format:** `.TAP` — **identical** to the 48K Spectrum, so
  `../zxspectrum/tapfile.ts` (`buildTap`/`parseTap`) and the cassette codecs in
  `../zxspectrum/audio/` are reused unchanged.
- **ROM:** `public/roms/zxspectrum128.rom` — the **32 KB** 128K ROM image
  (ROM 0 = the 128 editor/menu, ROM 1 = the 48 BASIC ROM; bit 4 of `0x7FFD`
  selects between them). Distributed under Amstrad's long-standing permission for
  Sinclair ROM images, exactly like the existing `zxspectrum.rom`; add an entry
  to `public/roms/ATTRIBUTION.md`. _(Do not commit a fabricated ROM.)_
- **Reuse:** from `src/dialects/zxspectrum/` — `charset`, `keywords` (the 48K
  base set), `tapfile`, the `language` builder pattern, the keyboard matrix
  (`emulator/keyboard.ts`) and `keyboardLayout`, `vars` / `reports` / `sysvars` /
  `numbers`, and the cassette `audio/` codecs. Plus `src/emulator/z80/` (the
  vendored Z80 core — **use, never edit**) and the generic editor/keyboard seams.
- **License note:** the Z80 core is MIT (no new exposure). The 128K ROM ships
  under the same Amstrad permission as the 48K ROM already in the repo.

## A note on the `createEmulator` contract

`Dialect.createEmulator({ rom, ramKb })` types `ramKb` as `16 | 32 | 64`. The
128K **ignores `ramKb`** and always allocates its eight 16 KB banks internally —
exactly as `SpectrumMachine` already ignores it. **No change to
`src/dialects/types.ts` is required**, keeping the work confined to the dialect
folder.

## Stage 1 — Language core ✅

128 BASIC is the 48K Spectrum's Sinclair BASIC plus the two tokens the 48K table
deliberately omits. Text ↔ tokenized program bytes; no emulator, no registry
change.

- [x] `keywords.ts` — append the **128-only tokens** to the 48K set:
      `SPECTRUM` (`0xA3`) and `PLAY` (`0xA4`). The stub currently re-exports the
      48K `spectrumKeywords` verbatim; replace it with
      `[...spectrumKeywords, SPECTRUM, PLAY]` plus signatures/docs (`PLAY a$[,b$]`
      — play music strings on the AY chip; `SPECTRUM` — return to 48 BASIC).
- [x] `charset.ts` — **reused** (`spectrumCharset` re-export); the character set
      is identical. No work unless a 128-specific glyph surfaces.
- [x] `tokenizer.ts` / `detokenizer.ts` — handle the two extra tokens.
      `../zxspectrum/tokenizer.ts` / `detokenizer.ts` are now **parameterized by
      keyword table** (default = `spectrumKeywords`, backward-compatible); the
      128 stubs are thin bindings that pass `spectrum128Keywords`. Derived
      matcher/token tables are memoized per keyword array. Collects
      `TokenizeError[]`; does not throw.
- [x] image builder — **reused**: `tokenize` wraps the bytes in a `.TAP` via the
      re-exported `buildTap`; `detokenize` parses with `parseTap` (`index.ts`,
      unchanged).
- [x] `lint` wired through `tokenize` (in `index.ts`, unchanged).
- [x] tests: tokenizer round-trips a `PLAY`/`SPECTRUM` program; 48K-compatible
      programs tokenize to identical bytes (regression guard); the 48K tokenizer
      still rejects `PLAY`/`SPECTRUM` (proves the split).

**Depends on:** the `Dialect` contract only.
**Verify:** `npm test` + `npm run typecheck`.

## Stage 2 — Emulator core (paging + dual ROM) ✅

The heart of the 128K work. Extend the 48K machine wiring rather than forking it.

- [ ] `emulator/memory128.ts` — `Spectrum128Memory`: eight 16 KB RAM banks +
      two 16 KB ROM banks. Decode **port `0x7FFD`**: bits 0–2 = RAM bank paged at
      `0xC000`, bit 3 = displayed screen (bank 5 vs shadow bank 7), bit 4 = ROM
      select (0 = 128 editor ROM, 1 = 48 BASIC ROM), bit 5 = paging-disable lock.
      Bank 5 stays at `0x4000`, bank 2 at `0x8000`. Expose a `read`/`write`/
      `readWord`/`writeWord` surface compatible with `../zxspectrum/vars.ts` and
      `display.ts` so those reuse unchanged. Validate the ROM is **32768** bytes.
- [ ] `emulator/spectrum128Machine.ts` — a `Spectrum128Machine` implementing
      `MachineEmulator`, mirroring `SpectrumMachine`: one IM1 interrupt/frame,
      keyboard + border on port `0xFE` (reuse `../zxspectrum/emulator/keyboard`),
      `0x7FFD` writes routed to the memory, AY register select/data on
      `0xFFFD`/`0xBFFD` routed to `emulator/ay.ts`.
- [ ] **ROM-aware LD-BYTES flash-load trap.** The `0x0556` tape-loader trap must
      only fire when the **48 BASIC ROM (ROM 1)** is paged in (`0x7FFD` bit 4 = 1),
      since `0x0556` is meaningful only in that ROM; otherwise the same PC in the
      128 editor ROM is unrelated code. Gate the trap on the current ROM bank.
- [ ] **128K boot + load path** in `loadProgram`: boot to the **128K menu**, drive
      the menu to **"128 BASIC"** (highlight + ENTER) — verify the menu cursor
      sysvar / screen rather than assuming a fixed key count — then type
      `LOAD ""` and `RUN` through the key matrix exactly as the 48K machine does.
      (A simpler first cut may select **"48 BASIC"**, which collapses to the
      existing 48K path; the 128 menu route is what unlocks the extra RAM, `PLAY`
      and long programs and is the real target.)
- [ ] `emulator/display.ts` — **reused** width/height; render the **active**
      screen bank (5 or 7 per `0x7FFD` bit 3) through `../zxspectrum`'s
      `renderDisplay`.
- [ ] `emulator/ay.ts` — AY-3-8912 register file that **absorbs** the I/O so
      `PLAY`/`BEEP` run without error. Audio _output_ is **not** in the
      `MachineEmulator` contract today, so Stage 2 needs only the register file;
      real sound is Stage 5 (see the contract-gap note there).
- [ ] `currentLine` / `debugStep` / `readVariables` / `readReport` — wire to the
      reused `../zxspectrum` `sysvars` / `vars` / `reports` over the 128 memory
      (the BASIC program and sysvars live in the standard 48K-compatible map).
- [ ] ROM into `public/roms/zxspectrum128.rom` (32 KB) + `ATTRIBUTION.md` entry.
- [ ] test: boot the real `public/roms/zxspectrum128.rom` (skip cleanly if
      absent), reach the menu, enter 128 BASIC, inject and run a `PRINT` program,
      assert on display memory; a second test exercises a `PLAY`/paging program.

**Depends on:** Stage 1 (charset for display, image builder for `loadProgram`).
**Verify:** emulator boot test passes — confirmed against the real 32K
`public/roms/zxspectrum128.rom`. The boot path drives the 128K menu to
"128 BASIC" (reading the highlighted item off-screen), types `LOAD ""` to
flash-load through the ROM-gated `0x0556` trap, then types `RUN` as a direct
command. The 128 full-screen editor renders its menu/listing with the 48 BASIC
font (file offset `0x7C00`) and redraws the listing on every keypress, dropping
keys typed mid-redraw — so `RUN` is typed with a wide inter-key gap. Paging,
dual-ROM, AY register-file and shadow-screen logic also have ROM-free unit tests
(`memory128.test.ts`, `ay.test.ts`).

## Stage 3 — Wire-up: keyboard + samples + register ✅

- [ ] `keyboardLayout.ts` — **reused** (the matrix is identical to the 48K). The
      stub re-exports `spectrumKeyboardLayout`; Stage 3 may clone it under a
      `zxspectrum128` id/theme if a distinct look is wanted (the +2/+3 had proper
      typewriter keys, but the matrix and key tokens are unchanged).
- [ ] `samples/` + `samples.ts` — canonical `hello` / `circles` / `breakout` /
      `maze` in Spectrum BASIC (`hello` is the starter), plus a 128-flavoured one
      showcasing `PLAY` and/or BRIGHT/FLASH colour. Each must tokenize cleanly and
      `hello` should actually run on the ROM.
- [ ] finalize `aiProfile.ts` — teach **128 BASIC** specifics: same Sinclair
      BASIC as the 48K (`:` multi-statement lines, full-screen editor, BIN/IN/OUT,
      colour via INK/PAPER/BRIGHT/FLASH/OVER/INVERSE) **plus** `PLAY` for AY music,
      the larger program space, and the 128 menu/RAMdisk commands; note `SPECTRUM`
      drops back to 48 BASIC.
- [ ] `index.ts` — already assembled against the stubs; confirm `createEmulator`
      returns `new Spectrum128Machine({ rom })` and `debuggable: true` holds.
- [ ] **register `zxspectrum128` in `src/dialects/registry.ts`** (next to
      `zxspectrum`).
- [ ] optional `.vk-theme-zxspectrum128` block in the keyboard CSS if re-themed.
- [ ] tests: keyboard-layout validation, samples tokenize cleanly, the live
      `createEmulator()` seam boots and runs.

**Depends on:** Stages 1–2.
**Verify:** `npm run typecheck` + `npm test` + `npm run dev` smoke + `npm run e2e`.
